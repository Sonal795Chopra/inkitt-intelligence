01 — The Problem

Signal is buried in noise

Inkitt runs three teams — Product, Data Science, and Content — each doing daily standups. The output is unstructured text: who said what, what they're blocked on, what they committed to. Useful signal exists, but it requires manual extraction to act on.

The CEO problem is specific: by the time cross-team blockers surface in a weekly sync, days have already been lost. A DS engineer waiting on a Product decision for 3 days is not visible unless someone manually connects the dots.

🔍

Hidden Blockers

Cross-team dependencies are invisible in per-team standups. No one owns the connection.

⏱

Delayed Signal

OKR erosion from blocked engineers accumulates silently before reaching leadership.

📋

Manual Overhead

Synthesising 3 transcripts into an actionable brief takes 30–60 mins of human effort daily.

02 — The Solution

A three-stage LLM pipeline

Each stage has a narrow, well-defined job. The output of each stage is structured JSON that feeds directly into the next. This makes the pipeline composable, debuggable, and easy to tune independently.

Stage 1 — Extract
Per-team extraction
Runs in parallel across all three teams. Each call reads one transcript and returns a structured object: every person, what they're working on, their blocker type, days blocked, and OKR tag.
→
Stage 2 — Match
Cross-team matching
Takes all three extracted objects together. Identifies dependency pairs — where person A is blocked waiting on person B from another team — and scores them by severity, days unresolved, and OKR impact.
→
Stage 3 — Synthesise
CEO digest generation
Takes the match output plus all three extractions. Produces a prioritised CEO brief: ranked items with situation, required action, owner, due date, and company-level pulse metrics.

"Stages 1 run in parallel — all three team extractions fire simultaneously — so total runtime is roughly the slowest single extraction, not the sum of all three."

Stage 1 — Extraction prompt design

The extraction prompt instructs the model to read one standup transcript and return a structured JSON object. The key design choice is the blocker taxonomy: rather than a free-text blocker field, the prompt constrains output to four types. 

// Blocker type taxonomy — constrains LLM output
"waiting_known"   → blocked, know who can unblock, cross-team
"waiting_unknown" → blocked, unclear ownership
"fundamental"     → technical/design problem, no clear unblocking path
"at_risk"         → not blocked yet, but commitment is at risk

This taxonomy matters because Stage 2 (matching) only needs to look at waiting_known blockers — those are the cross-team dependencies. The other types inform the dashboard but don't drive the matching logic.

Stage 2 — Cross-team matching

The matcher receives all three extraction objects in a single prompt. It identifies pairs where one person's waiting_on_team and waiting_on_person corresponds to a real person in another team's output. 

// Each match object — output by Stage 2
{
  "match_id":           "m1",
  "type":               "cross_team_dependency",
  "severity":           "critical",     // critical | high | medium
  "person_a":           "Jin",           // the blocked person
  "team_a":             "datascience",
  "person_b":           "Marcus",         // the unblocking person
  "team_b":             "product",
  "days_unresolved":    4,
  "okr_impact":         "Experiment Velocity — A/B framework delayed",
  "recommended_action": "Marcus to share API spec with Jin by EOD",
  "silent":             false
}

The silent flag identifies blockers that neither person has raised in their standup — discovered by cross-referencing commitments. These are flagged separately because they represent invisible organisational debt.

Stage 3 — CEO digest synthesis

The digest prompt receives the full matcher output plus all three extractions. It is instructed to write for a CEO audience: no technical jargon, prioritised by urgency, with one clear action per item, and a named owner. 

// Each digest item — output by Stage 3
{
  "priority":   1,
  "status":     "🔴",
  "title":      "A/B framework blocked — 4 days",
  "situation":  "Jin (DS) cannot run Experiment Velocity tests...",
  "how_long":   "4 days unresolved",
  "action":     "Marcus shares API spec with Jin by EOD",
  "owner":      "Marcus (Product)",
  "due":        "today",
  "okr_impact": "Experiment Velocity"
}
03 — The Interface

Intelligence tab — three panels

The main view is three panels in a fixed horizontal layout. Each panel has a distinct job: input, analysis, and action.

① Transcripts
② Team Intelligence
③ CEO Digest

① Transcript Input

Tabbed input for all three teams. A green/teal dot appears on each tab once content is loaded. The Run Analysis button activates only when all three are populated, preventing incomplete analysis.

② Team Intelligence

Per-person rows with blocker severity chips. Cross-team and CEO escalation badges are computed client-side by cross-referencing the matcher output — no extra LLM call needed.

③ CEO Morning Digest

Priority-ranked items with status, situation, action block, owner, and due date. Below the items: company pulse with days lost, on-track teams, at-risk teams, and shared bottleneck strings.

Progressive Loading

The status bar in the header shows which phase is running. Extractions fire in parallel — each updates its team card as it completes, so the UI populates incrementally rather than all at once.

04 — Commitments Ledger

Commitments tab — promise tracking across teams

The second tab is a live commitment ledger: every promise made in the standup, tracked by status, owner, due date, and downstream impact. It answers the question the digest doesn't: who committed to what, and is it slipping?

"The ledger is built entirely client-side from the extraction and matcher outputs — no extra LLM call. buildCommitmentLedger() is a pure function that derives status, days delta, and downstream impact deterministically."

Commitment object shape

Each commitment is a flat object derived from Stage 1 extraction output, enriched with matcher data:

// Each commitment in the ledger
{
  "id":          "commit_001",
  "owner":       "Ren",
  "team":        "Content",
  "what":        "Getting started guide first draft",
  "due_date":    "2026-03-25",      // parsed from LLM freeform ("by Friday", "EOD", etc.)
  "status":      "slipped",          // open | due_today | slipped | closed
  "days_delta":  -5,                 // negative = overdue
  "okr_tag":     "OKR 1.2",
  "at_risk":     true,
  "downstream":  ["CEO"],             // who is blocked waiting on this person
  "source":      "standup"           // standup | seeded
}

Status logic

Status is computed deterministically from due_date relative to today, with no LLM involvement: 

// computeStatus(dueDate, atRisk)
delta === 0           → "due_today"
delta > 0            → "open"
delta < 0            → "slipped"   // any overdue = slipped

// Date parser handles freeform LLM output:
"EOD" / "today"      → today's date
"March 25"           → 2026-03-25
"Mar 25th"           → 2026-03-25
"25 March"           → 2026-03-25
"2026-03-25"         → passthrough
"by Friday" etc.     → null → status defaults to "open"

Downstream cross-reference

The downstream field is built by joining the commitment ledger with the Stage 2 matcher output. For every cross-team match (person_a blocked → person_b), person B's commitment row gets person A in its downstream array. If that match was also CEO-escalated, "CEO" is injected instead — making CEO-impacting slips instantly visible in the ledger.

Three UI sections

① Summary Bar

Four metric cards: Total Open, Due Today (gold highlight when non-zero), Slipped (coral highlight when non-zero), and Hit Rate %. Hit rate = closed ÷ (closed + slipped).

② Commitment Ledger

Full table sorted: slipped (most overdue first) → due today → open (soonest first) → closed. Slipped rows get a coral left border. Status, owner, team, commitment text, due date, days delta, OKR tag, and downstream all shown.

③ Team Health

Per-team row with open/due-today/slipped counts, hit rate %, and an inline health bar (teal = closed proportion, coral = slipped proportion). Makes cross-team commitment reliability visible at a glance.

Seeded historical data

To make hit rate meaningful from day one, the ledger seeds 12 historical items: 7 closed and 5 slipped, distributed across teams to produce realistic per-team rates. This gives the metric weight even on the first run — without it, 0 closed / 0 resolved would render the hit rate blank.

05 — The Dashboard

Org Dashboard — six analytical views

The third tab is an ops-level dashboard built from the same extraction data. It doesn't make additional API calls — all six views derive from the three extraction objects plus the matcher output.

  • 1
    OKR Progress with Erosion Model

    Five hardcoded company OKRs, each with a progress percentage. An erosion overlay shows how many OKR points are being lost due to active blockers, calculated as days_blocked × 1.8 pts, capped at 35% of progress.

  • 2
    Team Health Sparklines

    5-point health trend per team: 4 historical data points (hardcoded from prior standups) plus today's live extracted health. Green → 3, Amber → 2, Red → 1. Shows trend direction: improving / stable / degrading.

  • 3
    Blocker Dependency Map

    Force-directed SVG with three team nodes. Edges are drawn between teams with active cross-team blockers — thickness scales with days unresolved, colour indicates severity. Teams flagged as shared bottlenecks get a pulsing ring animation.

  • 4
    Commitment Hit Rate

    16-person rolling hit rate (6-sprint average). Sorted ascending to surface underperformers. Coral = below 60%, gold = 60–75%, teal = above 75%. People below threshold get a warning line.

  • 5
    Action Board

    Table of required actions derived from the matcher. CEO escalations appear first with an urgent triangle marker. Each row: action text, owner, due date, who is unblocked, and which OKR is affected.

  • 6
    Impact Calculator

    One card per critical OKR impact. Shows a three-zone bar: healthy progress, current erosion (red solid), and projected erosion in 5 more days (red dashed). The large number shows days already lost.

06 — Technical Stack

What it's built with

Claude claude-opus-4-6 API React 19 TypeScript Vite 8 Tailwind CSS v4 GitHub Pages SVG (dependency map) Google Fonts (Playfair Display)

The three prompt files (extraction.js, crossTeamMatch.js, ceoDigest.js) each export a single async function that calls the API and returns parsed JSON. This keeps the prompt logic fully decoupled from the React components.

The entire app is a static build with no backend. The Anthropic API key is entered at runtime via a gate screen on first load and stored in sessionStorage only — it is never baked into the bundle, never committed to the repository, and cleared automatically when the tab closes. A production version would proxy calls through a backend to remove the key from the browser entirely.

"All API calls use structured JSON output — the system prompt specifies a strict schema, and the response is parsed with JSON.parse(). If the model returns invalid JSON the UI shows an error state."

07 — Design Decisions

Key choices and why

Decision Rationale
Three separate LLM calls vs. one big call Separation of concerns. Each stage can be tuned, replaced, or debugged independently. A single monolithic prompt extracting + matching + synthesising is harder to reason about and iterate on.
Parallel extraction (Promise.all) The three team extractions have no dependency on each other. Running them in parallel cuts total runtime from ~45s to ~15s. Each result updates the UI immediately as it resolves.
Structured JSON output (not free text) Every downstream component — badges, chips, bars, maps — needs machine-readable data. A structured schema means the UI can render deterministically without parsing natural language.
Blocker taxonomy (4 types) Free-text blockers are too vague to act on. Constraining to 4 types forces the model to classify severity and enables the matcher to filter precisely for cross-team dependencies.
CEO-specific digest stage The extraction and matching outputs are dense and technical. A separate synthesis stage — prompted specifically for CEO reading level — produces language that is immediately actionable without translation.
OKR erosion calculated client-side Rather than asking the LLM to estimate OKR impact (which varies by interpretation), a simple formula — days × 1.8 pts — is applied deterministically. Consistent and fast.
Commitment status computed client-side, not by LLM Asking the LLM to decide if a commitment has slipped introduces hallucination risk and latency. Instead, computeStatus() compares the parsed due date against today's date deterministically. The LLM only provides the raw date string; the status logic is pure JavaScript.
Seeded historical data in the ledger Without prior standups, hit rate is always null or 0%, making the metric useless on first use. Seeding 12 realistic closed/slipped items gives the chart immediate meaning and shows per-team differentiation (Product green, DS amber, Content red) from the first analysis run.
Dependency map as SVG (not a library) The graph has exactly 3 nodes in fixed positions. Using D3 or a graph library would add significant bundle weight for a layout that can be hardcoded. Hand-drawn SVG paths give full control over aesthetics.

"The goal was a working prototype that could run a real Inkitt standup and surface something genuinely useful — not a polished product. Every decision was made to maximise signal clarity, not feature count."
